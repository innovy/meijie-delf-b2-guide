## 先说结论

这个项目已经改为自定义后台，不再使用 Decap CMS。前台和后台都围绕 `content/data.json` 工作：作者卡、栏目、文章、排序、标签、摘要和正文都存在同一份结构化数据里。

部署到 Netlify 后，访问 `/admin/` 就能进入后台。线上保存通过 Netlify Function 写回 GitHub。

## 从飞书内容迁移

你现在内容在飞书里，可以先按文章拆分：

1. 每个飞书页面对应一篇攻略文章。
2. 标题放到“标题”字段。
3. 摘要用 1-2 句话写清楚这篇文章解决什么问题。
4. 正文从飞书复制到后台编辑器里。
5. 图片不要复制路径，直接在后台点图片按钮上传。

## 图片怎么可视化添加

后台正文工具栏里有“插图”按钮。选择本地图片后，后台会自动插入图片语法，并在保存时把图片上传到 `assets/uploads/`。

如果以后文章图片非常多，可以再升级到 Sanity、Strapi 或 Directus，这类后台更适合做图片库、音频、视频和小程序 API。

## 双向链接怎么写

用两个方括号写关键词：

```markdown
[[写作]]
[[考试结构与评分]]
[[口语|B2口语表达]]
```

第一种会显示文章标题，第三种会链接到“口语”文章，但显示成“B2口语表达”。页面底部会自动出现“这些文章也提到本主题”。

这部分目前仍建议手写，因为它是知识库内部链接的核心能力。好消息是格式很短，只要记住两个方括号。

## 音频和视频预留

MVP 暂时不放音视频，但已经预留了写法：

```markdown
::media{type="audio" src="assets/listening-sample.mp3" caption="听力片段示例"}
::media{type="video" src="assets/oral-demo.mp4" caption="口语示范视频"}
```

正式上线前可以再加转码、封面、字幕和访问权限。

## 后台具体操作步骤

1. 把项目上传到 GitHub。
2. 用 Netlify 从 GitHub 导入这个仓库。
3. 在 Netlify 的 Environment variables 里配置 `ADMIN_PASSWORD`、`GITHUB_REPO`、`GITHUB_BRANCH`、`GITHUB_TOKEN`。
4. 打开 `你的网址/admin/`。
5. 输入后台密码，编辑内容。
6. 点击“保存到线上”，后台会把 `content/data.json` 提交回 GitHub。
7. Netlify 监听到 GitHub 更新后会自动重新部署，前台页面会更新。

## 自动摘要、标签和本文目录

后台里的“摘要”和“标签”可以手动填写，也可以留空。

- 摘要留空时，前台会读取正文第一段，自动截取成文章摘要。
- 标签留空时，前台会根据标题、栏目和正文关键词自动生成，例如写作、口语、报名、评分、备考。
- 本文目录不需要手动写。正文里只要有 `## 二级标题` 或 `### 三级标题`，文章顶部就会自动生成“本文目录”。

手动写的摘要和标签优先级更高。也就是说，自动生成适合快速发布，手动填写适合后期精修。

## 栏目、排序和网址后缀

后台里现在有三个主要入口：

- 栏目管理：新增、删除、修改栏目，并用“栏目排序”控制左侧一级目录顺序。
- 攻略文章：新增、编辑文章，并用“排序”控制同一栏目下的文章顺序。
- 作者卡/站点：修改网站标题、作者姓名、头像、简介、微信号和二维码。

“网址后缀”就是文章地址里 `#` 后面的唯一名字，例如 `#writing`。它的作用是让文章链接稳定，方便双向链接和外部分享。普通编辑时可以留空；发布后如果已经有人收藏或引用这篇文章，就不要频繁修改。

自定义后台已经支持拖动栏目和文章排序。飞书粘贴整理是实用版：可以自动识别标题、正文、摘要、标签；飞书里的复杂表格、数据库、评论和权限内容仍建议在后台里二次检查。

## “这些文章也提到本主题”是什么意思

这是双向链接带来的反向目录。比如 A 文章里写了 `[[写作]]`，它会链接到“写作”这篇文章；当你打开“写作”文章时，页面底部就会显示“这些文章也提到本主题：A 文章”。

它的作用不是普通推荐，而是告诉读者：这个主题还在哪些文章里被引用过。这样学生可以从一个知识点反向找到相关学习路径。

举例：

- [[备考路线图]] 里提到 [[写作]]。
- 打开“写作”文章时，底部会看到“备考路线图”。
- 这说明“写作”不仅是一篇专项文章，也出现在整体备考规划里。

## Hugging Face 适合做什么

Hugging Face Static HTML 很适合测试展示这个网页，因为它部署静态文件很方便。但它不是面向内容编辑的 CMS 后台；你可以展示网站，但不适合作为“老师日常可视化更新文章”的主后台。

## 小程序路线

先把文章结构、栏目、双链和图片管理方式固定下来。将来做微信小程序时，最好让小程序读取同一套内容 API，不要再维护第二份内容。
